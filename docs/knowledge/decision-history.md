## 决策详情

### 决策 1: 采用精简版目录结构

**日期**: 2026-04-09

**来源**: 001-governance-foundation

**决策内容**:
项目初期采用精简版目录结构，只包含核心文件。

**理由**:
- 第一阶段最小可用原则
- 避免过早设计过多结构
- 后续按需扩展

**替代方案**:
- 方案 A: 完整目录结构（过早复杂）
- 方案 B: 扁平结构（不利于扩展）

**影响范围**:
- openspec/ 目录结构
- docs/ 目录结构
- .superpowers/ 目录结构

---

### 决策 2: Dogfooding 验证为主、修复为次

**日期**: 2026-04-10

**来源**: 002-workflow-dogfooding

**决策内容**:
002 feature 的目标是验证治理框架，发现缺口先记录，不立即修复。

**理由**:
- 避免 feature 范围失控
- 保持验证的客观性
- 产出清晰的问题清单

**替代方案**:
- 方案 A: 发现问题立即修复（范围不可控）
- 方案 B: 只记录不产出方案（无后续输入）

**影响范围**:
- 002 feature 的执行方式
- 问题清单的产出

---

### 决策 3: 不创建缺失知识文件

**日期**: 2026-04-10

**来源**: 002-workflow-dogfooding

**决策内容**:
发现 docs/knowledge/ 下文件缺失，但不在此 feature 创建。

**理由**:
- 遵循"验证为主"原则
- 创建文件会偏离 dogfooding 目标
- 留给下一个 feature 处理

**替代方案**:
- 方案 A: 立即创建（偏离目标）
- 方案 B: 只在 knowledge-index.md 中标注（不够完整）

**影响范围**:
- 知识层文件结构
- 003 feature 的范围

---

### 决策 4: 渐进式修复问题

**日期**: 2026-04-10

**来源**: 003-feature-hardening

**决策内容**:
将 002 发现的问题分到多个 feature 修复，而非一次全部解决。

**理由**:
- 分阶段降低风险
- 每个 feature 聚焦一个领域
- 便于验证和回滚

**替代方案**:
- 方案 A: 全部在 003 修复（范围过大）
- 方案 B: 只修复高优先级（不完整）

**分配**:
| Feature | 修复内容 |
|---------|----------|
| 003 | 知识文件、状态规则、模板扩展 |
| 004 | skill 调用方案 |

**影响范围**:
- 003 和 004 feature 的范围定义

---

### 决策 5: 状态同步规则嵌入模板

**日期**: 2026-04-11

**来源**: 005-template-refinement

**决策内容**:
将状态同步规则直接嵌入 tasks.template.md 和 runstate.template.md，而非只在 conventions.md 中维护。

**理由**:
- AI 读取模板时自动获取规则
- 减少查找成本
- 规则与模板结构紧密关联

**替代方案**:
- 方案 A: 只在 conventions.md 维护（分散，查找成本高）
- 方案 B: 创建单独的规则文档（增加文档数量）

**影响范围**:
- tasks.template.md
- runstate.template.md
- 后续 AI 执行效率
