## 4. 框架应用模式提炼

### 模式 1: Spec 决策固化模式

**发现场景**: 001-core-extraction spec 审阅

**模式描述**:
在 spec 定义阶段，发现模糊点（如字数定义、日期格式）时，不假设或猜测，而是：
1. 分析多种方案
2. 从可用性、可扩展性角度选择
3. 将决策固化到 spec 文件
4. 后续实现严格按决策执行

**效果**: 避免实现返工，团队对决策有共识

---

### 模式 2: 新项目框架复制模式

**发现场景**: md-meta-extractor 项目初始化

**模式描述**:
创建新项目时：
1. 复制 amazing-workflow 的模板文件
2. 根据项目特点调整 AGENTS.md 和 PROJECT_RULES.md
3. 保持 feature 生命周期不变
4. 立即可用框架流程

**效果**: 快速启动，无需重新设计治理结构

## 5. 对 amazing-workflow 的改进建议

### 5.1 高优先级

| 建议 | 原因 |
|------|------|
| 添加"决策固化流程"到 feature-lifecycle | spec 阶段需要明确澄清步骤 |
| 添加"新项目模板"到 templates | 快速复制框架结构 |

### 5.2 中优先级

| 建议 | 原因 |
|------|------|
| 添加自动化状态同步脚本 | 减少手动维护 |

| 建议 | 原因 |
|------|------|
| 多语言项目模板（Go、Python、Node） | 不同技术栈的目录结构差异 |
